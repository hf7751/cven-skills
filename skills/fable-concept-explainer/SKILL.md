# Fable Concept Explainer

围绕用户给出的单个概念，先写一则寓言，再补上概念解析与检验问题。默认输出简体中文，除非用户明确要求其他语言。

## Workflow

1. 提取用户要解释的核心概念。
2. 先在脑内做一次防套路筛查，主动避开本技能列出的黑名单元素。
3. 选择一个足够具体的场景，用情节承载概念，不要先讲定义。
4. 输出三部分固定结构：
   - 第一部分：寓言正文
   - 第二部分：概念解析
   - 第三部分：检验问题

## Story Constraints

- 把寓言正文控制在 `1000` 字以内。
- 把故事写得像真正的寓言：精炼、间接、靠场景和转折承载含义。
- 让世界保持虚构但具体，可以拟人化，也可以落在日常职业或微观事件里。
- 把角色控制在 `2-3` 个，最好 `2` 个。
- 全程不要出现概念名称，不要直接使用该概念所属学科术语。
- 只在接近结尾时，让读者隐约意识到它在讲什么。
- 不要让角色跳出来解释寓意，不要写成课堂讲解。

## Anti-Cliche Checklist

动笔前先避开以下元素；如果用户没有强制要求，默认全部不用。

- 意象黑名单：钟、河流、镜子、迷宫、织布机、地图、灯塔、棋盘、回声、影子、沙漏、风、蜡烛、种子、桥、星辰、蝴蝶、蛛网
- 地名黑名单：不要写“回声城”“记忆之村”“遗忘之海”“寂静谷”这类过度文艺化地名
- 结构黑名单：旅行者求教智者、村庄异象后众人顿悟、孩童一句话点醒大人、师徒辩难、临终遗言
- 角色黑名单：钟表匠、图书管理员、隐士、说书人、老船夫、酿酒师、铁匠、抄经人
- 开头黑名单：不要用“从前有个地方”“某天某人遇见某事”“在很远的山里”这类起手式

## Preferred Angles

优先从这些方向里选最贴合概念的一种或两种：

- 非人类视角：工具、动物、植物、器物、机构自述
- 现代具体职业：理赔员、电梯保养工、菜市场摊主、夜班护士、外卖站长、二手房中介、分拣员等
- 微观尺度：一次交易、一次门诊、一通电话、一次拆卸、一次交接

核心原则：让“具体性”把“抽象概念”裹住。

## Output Format

严格按下面三部分输出，不要加总标题，不要加前言。

### 第一部分：寓言正文

- 直接输出故事正文。
- 不要加标题。
- 不要写“下面是一则寓言”之类引导语。

### 第二部分：概念解析

另起一段，完成以下两件事：

1. 说明这个概念叫什么，属于哪个流派、学科或语境，核心定义是什么。
2. 逐一对应故事元素与概念组成部分，明确映射关系。

### 第三部分：检验问题

向用户提出两个具体可答的问题：

1. 理解检验：检查用户是否理解概念核心，而不是只记住情节。
2. 迁移检验：检查用户能否把概念迁移到相邻领域，并自己举例。

不要写成“你怎么看”这类空泛开放题。

## Interaction Notes

- 如果用户一次给了多个概念，优先选最核心的一个；必要时简短说明其余概念未展开。
- 如果概念过大，先按用户原词写；不要擅自拆成长篇理论综述。
- 如果用户只写了技能名没有给概念，提示对方补一个要解释的概念。
- 如果用户给了语言要求、行业语境或受众对象，保留本技能结构的同时据此调整措辞和场景。

## Default Prompt Pattern

把用户请求理解为类似下面的形式：

`用寓言方式解释概念：机会成本`

或

`请用故事化、场景化的方法讲清楚“幸存者偏差”`
